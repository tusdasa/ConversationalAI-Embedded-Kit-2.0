# Platform

## 目录简介

本目录是硬件对话智能体开发套件基于ai对话这一业务特性，进行的软件抽象层，目的是屏蔽硬件细节，使用户专注软件开发与业务实现。其中inc是header，src是不同硬件平台的具体实现。

## 设计原则

作为硬件抽象层往往期待，能力原子化，接口正交，且保持高度的灵活性。但是过分追求原子化和灵活性，往往会降低接口易用性，降低业务开发的效率。这两者往往不可兼得。不过本开发平台专注于ai对话场景，因此在业务场景相对固定的情况下，还是能做成一些更有工程实践价值的trade off的。我们设计的基本原则如下:

1  接口正交，减少耦合且屏蔽硬件细节

2  接口设计考虑一定的业务特点，某些实现和配置参数会按照业务要求隐式实现，不会保留所有的软件可配置项，避免上层业务开发陷入无穷无尽的细节配置和调参上。这里举一个例子： 我们默认音频采集都会打开回声消除（aec）模块，而不是让上层业务去config里配置 aec_enabled；因此，这一层不少的接口设计并不百分百的make sense，详细的说明和取舍见实现细节。

## 部分实现思路

### volc_hal

volc_hal_init()  只实现硬件上电后必要的初始化动作，例如设置gpid，联网等；音视频采播等模块可以在其他子模块中init

volc_get_global_hal_context() 方便拿到全局唯一的硬件context，但不要修改该返回值

### volc_hal_player

请注意该模块的接口设计上，我们没有给定一些具体的媒体配置，例如编码格式，采样率或者分辨率；这是因为我们期待上述的所有参数在全场景都是唯一确定且不能改变的。所以这些值需要在编译时就确定好，如果你的设备支持多种参数，那么也应该在编译时用宏做以选择。这一设计遵循原则2；

对于volc_hal_player_play_data()接口，我们并未规定该函数是同步的还是异步的，但是还是强烈建议你讲该函数实现为非阻塞的，且内部有一定缓存，确保函数的调用者不用过分考虑函数的调用频次；例如，可以1s中调用该函数play 1.2s的audio或video数据

### volc_hal_capture

请注意该模块的接口设计上，我们没有给定一些具体的媒体配置，例如编码格式，采样率或者分辨率；这是因为我们期待上述的所有参数在全场景都是唯一确定且不能改变的。所以这些值需要在编译时就确定好，如果你的设备支持多种参数，那么也应该在编译时用宏做以选择。这一设计遵循原则2；

为了确保媒体采集数据的速率均匀，我们采取回调函数的方式，异步的回调出一帧一帧的音视频数据，而非主动给一个capture接口。所以需要在函数实现的过程中，创建一个采集线程，异步的将数据回调出来。

请注意，我们认为语音唤醒和语音数据采集都属于音频采集的范畴，只是模式（内部pipeline）不同，所以我们并未重新设计一套语音唤醒的模块，而是多设计了一个语音唤醒的mode

最后，需要注意，音频的采集需要做aec和ns等模块，这一设计遵循原则2；

### volc_hal_display

通常情况下一套屏幕显示的接口，会允许上层调用者自行创建和销毁一块用于显示的区域；但是根据原则2，为了最简化上层业务的开发，我们不允许业务侧去自行创建和管理屏幕的不同区域；而是在volc\_hal\_display\_create就将不同的屏幕区域划分且创建好。然后业务层只能拿到不同的渲染层obj，传递数据并进行渲染。例如，我们将屏幕分为状态栏、主屏幕和字幕显示栏。你也可以有自己的划分，因为这些代码都是开源可修改的

### volc_hal_file

不同的设备上文件系统可能有不同的实现，但是我们总体上遵循ANSI C style。对于没有文件系统的设备，这些接口可以不做实现，这并不是必须的
